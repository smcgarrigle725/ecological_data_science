# Schema Documentation

This directory contains the DynamoDB table definitions and design rationale for the microparticles_saint_john database.

## Table Schemas

### Site
The hub table. Every water, sediment, and animal sample links back to a site record via `siteid`.

| Attribute | Description |
|-----------|-------------|
| `siteid` | **Partition key.** Autogenerated: Year + Site + sample type (e.g. `2022_HarbourNorth_Animal`, `2022_HarbourNorth_Water`, `2022_HarbourNorth_Sediment`) |
| `collection_organization` | Name of organization collecting the samples |
| `primarycontact` | Name of person at the organization to contact with questions |
| `year` | Year samples collected |
| `waterbody` | Name of waterbody where samples were collected |
| `waterbodytype` | `Freshwater`, `Marine`, or `Brackish` |
| `site` | Name of site |
| `latitude` | Latitude of site |
| `longitude` | Longitude of site |
| `substrate` | Description of substrate at the site |
| `benthicpelagic` | `Benthic` or `Pelagic` |
| `tidaldepth` | For marine samples: `intertidal`, `subtidal`, etc. |
| `samplestaken` | Number of samples taken |
| `sampletypes` | `Animal`, `Sediment`, or `Water` (or combination) |

### FTIR
One record per individual MP particle processed by micro-FTIR spectroscopy. Links back to any sample table via `id` (the parent sample's partition key).

| Attribute | Description |
|-----------|-------------|
| `mp_id` | **Partition key.** Autogenerated: sample `id` + `_XX` (XX = MP number from that sample, e.g. `2022_HarbourNorth_Animal_S01_03`) |
| `id` | Foreign key → Animal / Sediment / Water (parent sample partition key) |
| `sampleid` | Sample identification number as given by collection group |
| `collection_organization` | Name of organization collecting the samples |
| `primarycontact` | Name of person at the organization to contact with questions |
| `date` | Date sample was sent to the FTIR lab |
| `common_plastic` | Common plastic? `Y`/`N` |
| `ftir_primary` | Primary MP material identified |
| `ftir_secondary` | Secondary MP material |
| `ftir_tertiary` | Tertiary MP material |
| `ftir_quarternary` | Quaternary MP material |
| `ftir_quinary` | Quinary MP material |
| `cellulosic_peak` | Cellulosic peak present? `Y`/`N` |
| `ftir_size_microns` | MP size determined by lab (microns) |
| `colour` | Primary MP colour |
| `other_colour` | Secondary MP colour(s) |
| `structure` | MP structure: `fibre`, `fragment`, `sphere`, or `film` |
| `filename` | Name of spectrogram file provided by the FTIR lab |

## Key Design Notes

**Primary keys use autogenerated composite IDs.** The `siteid` and sample `id` fields are constructed from year, site name, sample type, and sample number. This makes records human-readable and traceable back to their source without needing a separate lookup, while ensuring uniqueness across partner organizations.

**`id` vs `mp_id` in the FTIR table.** Each FTIR record represents a single MP particle. `mp_id` is the FTIR table's partition key. The parent sample is identified by `id`, which matches the partition key of the Animal, Water, or Sediment record — this is the join key between FTIR and the sample tables. Note that the FTIR table does not include a `sample_type` field: MPs sent for spectroscopic analysis were selected randomly across samples and the originating sample matrix was not recorded at the time of submission. As a result, FTIR records can be joined to a parent sample by matching `id`, but the matrix type (animal, water, or sediment) must be inferred from the structure of the `id` string if needed.

**Matrix-specific fields are intentionally separated.** `samplevolume` (water), `sampleweight_wwt` (sediment), and `animallength` / `tissuetype` / `number_animal` (animal) only appear in their respective tables. Keeping these in separate tables avoids sparse records and makes each table's schema self-documenting for partner data contributors.

**GSIs on `siteid`** in the Animal, Water, and Sediment tables allow efficient retrieval of all samples at a given site without scanning the full table.
